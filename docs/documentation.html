Overview

What this is

There's no purpose-built, Jain-specific place-finder — generic map apps either don't know about these places or bury them under irrelevant results. This app searches OpenStreetMap's live dataset for Jain community places near any location the user searches or their current position, filterable by category, with no account, install, or cost to the user.

It runs entirely on free infrastructure: static hosting plus public map-data APIs. Any monetization (a voluntary donation link) exists only to offset the maintainer's real out-of-pocket costs, not to generate profit.

Features

  • 🔎 Search by typed place name or device geolocation
  • 🛕 Filter by category — Temple/Derasar, Dadabari, Upashray/Sthanak, Panjrapole/Goshala, or all combined
  • 🪷 Denomination badges (Digambar / Shwetambar / Sthanakvasi) shown where source data has them
  • 📏 Adjustable search radius, 5–100 km
  • 🗺️ Map and distance-sorted list, with per-category icons
  • 🧩 Graceful degradation — if one category's search times out, the others still return results
  • 🧭 Directions to any result via Google Maps
  • ✏️ Report a correction, deep-linked into OpenStreetMap's own editor
  • 🔗 Share a link to one specific place — renders instantly, no extra lookup
  • ⭐ Save places to a personal list (no account), in a "Saved" tab
  • 🍽️ Look up nearby vegetarian food or Dharamshalas from any result
  • 📲 Installable to your home screen, with the app shell cached for quick loads
  • 🙏 Voluntary UPI donation link in the footer
  • ➕ Planned: community "+ New" submissions via a live database
  • 🚧 Not yet designed: multi-stop route search, "verified by trust" badges

Business requirements

Objectives & scope

Objectives

  1. Make it fast and easy for a traveling Jain to find relevant places nearby, without an account or app install.
  2. Grow the map's coverage over time by letting the community contribute places that aren't yet mapped.
  3. Keep operating cost near-zero; any monetization only offsets real costs, never a profit target.

In scope — built

Search by name/geolocation, category filtering, denomination badges, map + list view, voluntary donation link.

In scope — planned

Community-submitted places via a "+ New" button, stored in a live database, visible to all users in real time and clearly marked as unverified community data, distinct from OpenStreetMap data.

Out of scope

User accounts, native mobile apps, an editorial review queue for submissions, trip planning / itineraries / reviews / bookings, paid tiers or feature paywalls.

Non-functional requirements

  • Cost: free-tier infrastructure only (GitHub Pages, public OSM APIs, Firebase free tier).
  • No login required for any current or planned feature.
  • Availability: best-effort — depends on third-party free services, no uptime SLA is realistic or promised.
  • Data licensing: all map data stays compliant with OpenStreetMap's ODbL license, attributed in-app.
  • Privacy: geolocation is used only client-side to run a search — never stored or sent to any server the maintainer controls.

Key design decisions

Case-sensitive regex, not ,i

Overpass's case-insensitive flag was dramatically slower and timed out at larger radii. Both capitalizations are spelled out in a plain alternation instead.

Per-category queries, not one union

An early version combined all categories into one Overpass query with many union branches and timed out under load. Each category now runs as its own smaller query, with graceful degradation.

No API keys in the current stack

Nominatim and Overpass are used key-free under fair-use policy — appropriate at this scale, worth revisiting if usage grows significantly.

Firebase config isn't a secret

The firebaseConfig object will be visible client-side by design — real access control comes from Firestore security rules, not from hiding the config.

Share links carry their own data

A shared place's name, category, denomination, and coordinates are encoded directly in the URL's query parameters, so opening a shared link renders instantly with no extra Overpass round-trip.

Favorites live in localStorage, not a database

Saving a place needs no account or network call. The tradeoff: saved places don't sync across devices — an acceptable cost for a personal list at this scale.

Service worker caches same-origin only

The fetch handler explicitly skips any request whose origin isn't the app's own, so map tiles, Nominatim, and Overpass always hit the network — caching those would risk stale search results.
